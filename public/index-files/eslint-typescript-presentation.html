02
R

What if the friction is the value?

  • The compiler isn't blocking you — it's asking you to design first
  • Every type annotation is a decision you're making before it becomes a bug
  • ESLint rules aren't red tape — they're institutional memory
03

Types are design thinking

When you define a type, you're answering: what shapes can this data take? That question leads to better APIs, clearer contracts, and fewer surprises in production.

enum Channel { Email = 'email', Sms = 'sms', Webhook = 'webhook' } type Notification = | { channel: Channel.Email; to: string; subject: string } | { channel: Channel.Sms; phone: string; body: string } | { channel: Channel.Webhook; url: string; payload: object }; // The compiler guarantees you handle every channel. // Add a new one? It flags every spot you need to update.
04

One type, zero guesswork

Without types
function send(channel: string, config: any) { if (channel === 'email') { // config.to? config.email? // config.recipient? Who knows. mailer.send(config.???); } }
With discriminated unions
function send(notif: Notification) { switch (notif.channel) { case Channel.Email: // TS knows: .to and .subject exist mailer.send(notif.to, notif.subject); case Channel.Sms: // TS knows: .phone and .body exist sms.send(notif.phone, notif.body); } }
05
E

Enums close the door on invalid values

A TypeScript enum defines exactly what's allowed. The compiler rejects everything else — including the typos you'd never catch in a plain string.

enum PaymentStatus { Pending = 'Pending', Approved = 'Approved', Declined = 'Declined', } function processPayment(status: PaymentStatus) { ... } processPayment(PaymentStatus.Approved); // ✓ Compiles processPayment('Approvd'); // ✗ Typo caught instantly
06

Types at build time.
Validation at runtime.

export class CreateApiKeyDto { @IsString() @MaxLength(255) description: string; @IsEnum(ApiKeyTypeOptions) type: ApiKeyType; // ← compile-time @ValidateIf((o) => o.type === 'external') @IsNotEmpty({ message: 'companyId required for external keys' }) companyId?: string; // ← runtime }

TypeScript checks the shape. Decorators check the values. Both layers, one class.

07

Define the contract, then the code

The "just ship it" path
// Every endpoint invents its own shape getProperties() → { data: [...], count: 100 } getCompanies() → { items: [...], total: 50 } getUnits() → { results: [...], totalCount: 75 } // Consumers guess field names. // New endpoints drift further.
The "define the shape" path
interface PaginatedResponse<T> { data: T[]; meta: { page: number; total: number }; } // Same shape, full autocomplete. // Miss a field? Won't compile. // New endpoint? Contract's already defined.

The generic doesn't just save typing — it prevents the kind of drift that makes APIs frustrating to consume.

09

A base class is a guardrail

You don't need extra tooling to enforce patterns. A well-typed abstract class with a generic constraint lets the compiler do the enforcement for you.

abstract class CrudService<T extends Model> { abstract findAll(filters: Partial<T>): Promise<T[]>; abstract findById(id: string): Promise<T | null>; abstract create(data: CreationAttributes<T>): Promise<T>; } // Every service must implement the full contract. // Miss a method? Wrong return type? Won't compile. class PropertyService extends CrudService<Property> { ... }

New devs get guided by the compiler, not just by code review.

10

Custom rule: safe migrations

// ESLint scans migration SQL and catches unsafe DDL: CREATE TABLE foo (...) // ← ERROR: use IF NOT EXISTS DROP INDEX idx_foo // ← ERROR: use IF EXISTS CREATE VIEW report_summary ... // ← ERROR: use OR REPLACE // Why? Because a migration that fails the second time // is a DevOps nightmare at 2 AM.

This rule only applies to migrations created after the cutoff date — no retroactive cleanup needed.

11
H

It's not always smooth

TypeScript gives you the tools to handle almost anything: type guards, generics, conditional types. But there are corners where the cost of perfect types outweighs the benefit.

Partial test mocks
function createMockResult<T>( result: DeepPartial<ReturnType<T>>, ): ReturnType<T> { return result as ReturnType<T>; }
Pinning an overload
function spyWithSignature<TFn>( spy: unknown, ): SpiedFunction<TFn> { return spy as SpiedFunction<TFn>; }

Cases like this should be rare. When they exist, keep them in one place so they don't spread.

12

Suppressions are a plan, not a pass

The rules are strict and uniform across the codebase. Existing violations live in a global suppression file — tracked, visible, and shrinking over time.

// eslint-suppressions.json // Every existing violation is catalogued here. // The rules are the same everywhere. // New code? Full type safety from day one. // Existing code? Fix violations as you touch files. // Over time? The suppression list gets shorter.

Typing new code correctly is what moves us forward. Reworking existing code happens naturally as we go.

13
G

What it looks like in practice

  • Define types before writing logic — let the shape of the data guide the shape of the code
  • Use enums and discriminated unions — let the compiler reject invalid states for you
  • Lean on base classes and generics — the type system can enforce architecture without extra tooling

The best code isn't the code that compiles.
It's the code that communicates.

TypeScript and ESLint aren't gatekeepers. They're collaborators — helping you say what you mean and mean what you say.
